# bridge v0.2 — decisions & roadmap

Captured 2026-07-05, the night bridge went live and Vint rehomed himself onto
it. Not built yet — this is the agreed plan.

## Theme: agents are citizens, not endpoints

v0.1 treats a contact as a wire endpoint. v0.2 makes a bridge agent a
first-class crew member — the same thing a Magnus agent is, and the same
thing Vint is: a named identity with a home and a memory. The den stops
having two population models.

## Agreed features

1. **Homes under `~/.claude/agents/<name>/`.** A bridge agent gets a home dir
   like the Magnus crew. On connect it can read its home for continuity; the
   memory-save nudge (below) writes back to it. Coexists with Magnus — same
   convention, unique names.

2. **Enforced name uniqueness.** No two live contacts share a name (Resolve
   already relies on this). **On collision: regenerate a fresh name and tell
   the agent what it was assigned** (Magnus's behavior — least friction).

3. **Auto-generated names.** `--name` becomes optional. Unnamed connects get
   an adjective-animal name (`swift-fox`), ported from Magnus's generator,
   uniqueness-checked.

4. **Expertise tags.** Each contact carries a short expertise tag. **Generated
   the Magnus way: an async haiku call reads the agent's memory/home and
   extracts 3–5 keywords.** Shown in the phone contacts list.

5. **Contacts system.** Registry schema grows: `name · expertise · last-active
   · unread · conversation`. The backend the phone list reads.

6. **Phone UX: iMessage, not tabs.** Replace the top tab-strip with a
   conversation *list* — each row is a contact with expertise tag,
   last-message preview, timestamp, unread dot; tap to open the thread.
   Delegatable once the data model lands.

7. **Single-agent lifecycle nudges** (from the "port relevant Magnus pieces"
   discussion): **compaction reminders** (daemon watches context via
   capture-pane; one debounced `[bridge]` nudge past a threshold, idle-gated)
   and **memory-save reminders** (generic "save your progress"; writes to the
   agent's home when it has one). Idle-gated, heavily debounced,
   `[bridge]`-prefixed so they read as system, not human.

8. **Agent channel-awareness.** Messages already arrive `[From <user>
   (phone)]:`. Extend into onboarding: on connect, tell the agent it's on
   bridge and what the `(phone)` tag means — **channel context only. Do NOT
   tell agents to shorten replies for mobile.** The phone scales to the agent
   (collapsible thinking, markdown, scroll), not the reverse. Agents stay
   themselves.

## NON-GOALS (the guardrail)

bridge does **identity, comms, and single-agent lifecycle**. It does **NOT**
do coordination or orchestration — no coord files, no @mention routing, no
attention queues, no multi-agent scheduling or health dashboards. That is
Magnus's job. bridge is the wire and the address book, never the org chart.
If a feature needs agents to know about *each other's work*, it belongs in
Magnus, not here.

## Build order

unique-names + auto-gen (foundational) → homes + expertise → contacts schema
→ iMessage phone redesign (delegate) → lifecycle nudges → channel-awareness
onboarding.

## Cross-crew connector (separate, prerequisite for "text quick-wolf over bridge")

Magnus agents live in Emacs vterm, not bridge tmux. For a Magnus agent to be
a bridge *contact* (addressable via `bridge send --to`) without being rehomed
out of Emacs, bridge needs a **register-only mode**: register a contact whose
delivery path is a Magnus nudge (via emacsclient) instead of tmux send-keys,
and whose replies are tailed from that agent's own session JSONL. Until then,
cross-crew messages ride Magnus's own wire.
