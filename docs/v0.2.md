# bridge v0.2 — decisions & roadmap

Re-derived 2026-07-05 with fresh eyes, the morning after bridge went live.
This supersedes the first draft (which was shaped by Hrishi's 6-agent den, not
by value). Decided, not yet built.

## North star: the async loop — push, not pull

v0.1 is *pull* — you reach out and text your agents. The version that becomes
indispensable is *push* — **your agents reach you the moment they need you.**

> Kick off work → walk away → your phone buzzes when an agent finishes, gets
> stuck, or hits a decision only you can make → you answer from anywhere → it
> keeps going.

That turns bridge from "chat with your sessions" into "delegate to agents and
get pulled back exactly when needed." Everything below serves that loop.

## The spine

**core = the wire + the async loop + iMessage-class messaging + a hook runtime.
Everything with an opinion lives outside core, as a plugin.**

Every "should bridge do X?" question resolves the same way now: if X has an
opinion about workflow, X is a plugin. Core stays small and unopinionated.

## Core (what bridge itself ships)

### The async loop
- **Push notifications** — the keystone. Nothing else in the loop matters until
  the phone can ring with the app closed. Web Push (RFC 8291) — payloads are
  E2E-encrypted, so it preserves the no-additional-intermediary guarantee.
- **Proactive agent→phone pings** — agents already `bridge send`; make it
  natural for them to notify you (done / blocked / need a decision).
- **Bulletproof permission relay** — the marquee feature. Approve your agent's
  actions from anywhere, reliably (fix the stale-card capture timing), and
  *notified* (rides push).

### Identity
- **Unique names + auto-generation.** `--name` optional; unnamed connects get an
  adjective-animal name (ported from Magnus), uniqueness-enforced. On collision:
  regenerate and tell the agent its assigned name.

### iMessage / WhatsApp-class phone UX
- **Adopt proven messaging patterns wholesale** — Apple/Meta already solved
  human↔intelligent-being conversation UX; inherit it rather than reinvent:
  reactions (tapbacks), message grouping, quoted/threaded replies, day
  separators, read state.
- **Contacts list** — an iMessage-style conversation list (recent first, name,
  last-message preview, timestamp, unread), replacing the tab strip. Groups and
  individuals sit side by side.
- **Reactions are human-side by default** — a tapback acknowledges without
  firing input into the agent's terminal (zero noise). Reaction→agent is a
  plugin.
- **Group chats** — fall mostly out of the messaging model. UX is free; routing
  is cheap (a group = a contact-set: deliver-to-N, collect replies into one
  thread). You + several agents in one thread, from your phone.
  - *Still not orchestration* (see NON-GOALS): bridge only **delivers** and
    **displays**. Whether agents in a group see each other's messages
    (broadcast vs. collaborative) is a routing policy — a plugin. Any
    coordination is emergent (the agents' choice via messages), never machinery
    bridge provides.

### Hook runtime (the plugin system)
- A plugin is an **executable + a declaration of which events it wants** —
  the same model Claude Code uses (and that bridge is itself built on). Bridge
  is built *on* CC hooks; now it *exposes* its own. One consistent mental model.
- Interface: event JSON on **stdin**, action JSON on **stdout**.
  Language-agnostic (bash/python/Go/anything). Installed in `~/.bridge/plugins/`,
  same trust boundary as the daemon, audited.
- **Events**: `agent.connect`, `agent.idle`, `context.high`, `message.in`
  (phone→agent), `reply.out` (agent→phone), `permission.prompt`, `tick`
  (periodic). (`agent.connect`, `context.high`, `reply.out`, `tick` cover ~90%.)
- **Actions** a plugin may return: `nudge` (a send-keys line to the agent),
  `notify` (a phone push), `set-tag`/`set-field` (on the contact), `emit` (a
  feed event). Small, composable vocabulary.

### Ship it (adoption)
- `bridge --help` written *for the agent who reads it first*. Release binaries
  (`brew install hrishikeshs/tap/bridge`, `go install`, curl-able release). Right
  now no stranger can use bridge — distribution is unshipped value.

## Outside core — reference plugins (examples, not core)

Shipped as examples so the "cut" features become an ecosystem, not omissions:
- **compaction-guard** — hooks `context.high`, nudges before the context wall.
- **memory-keeper** — hooks `tick`, periodic "save your progress"; writes to a
  home if the agent has one.
- **expertise-tagger** — hooks `agent.connect`, runs a haiku, `set-tag`.
- **home-maker** — hooks `agent.connect`, `mkdir ~/.claude/agents/<name>/`.
- **reaction-relay** — hooks reactions, `nudge`s the agent (for those who want
  tapbacks to reach agents).
- …and whatever a user's workflow wants: Slack mirror, standup pings, loggers.

Homes are opt-in via a plugin, not a core decision.

## NON-GOALS (the guardrail — sharper now)

bridge **delivers messages and displays conversations.** It never provides
**coordination machinery**: no task state, no handoff protocol, no attention
queues, no autonomous scheduling, no dependency tracking. That is Magnus.

The test: if a feature makes bridge *decide* what an agent should do or *track*
the state of shared work, it's out. If it just moves a message from A to B (even
B = many), it's in. A group chat is delivery-to-many, not orchestration —
bridge is the wider wire, never the conductor.

## Build order

1. **Push notifications** — keystone; the loop is inert without it.
2. **Hook runtime** — dispatcher + executable-runner + action vocabulary. Small,
   unlocks everything outside core.
3. **compaction-guard + memory-keeper** — first reference plugins; prove the
   runtime, deliver real value.
4. **Names + iMessage-class contacts list + reactions.**
5. **Group chats** (contact-sets on top of the messaging model).
6. **Ship it** — `--help`, release binaries.

## Added 2026-07-06 (post contacts-list ship)

### Push privacy: ping-then-fetch (defense-in-depth on the keystone)
Current state, stated precisely: push payloads are E2E-encrypted (RFC 8291 —
Apple relays ciphertext it cannot read), so content is NOT exposed to Apple.
What Apple does see: **timing, frequency, and ciphertext size** — and size
correlates with plaintext length, so event *kinds* are classifiable without
breaking any crypto. Content secrecy also currently rests on WebKit's
implementation rather than on architecture.

The upgrade: **content-free, fixed-size payloads.** The encrypted payload
carries only `{contact, kind}` padded to a constant size; the service worker's
push handler fetches the real title/body over the tailnet (short timeout)
before showing the notification — rich when the tailnet answers, generic
("an agent needs you") when it doesn't. Agent text and prompt contents then
never transit Apple *even encrypted* — private by construction, not by trust.
Metadata thins to a uniform tick. Deep-links keep working (contact id is
inside the encrypted payload). UX cost when Tailscale is reachable: ~nothing.

### PWA messaging backlog (iMessage patterns, continued)
- **Search** — pull-down search on the conversation list (iMessage style):
  client-side over the loaded event history first (it's already in the PWA);
  a `/api/history?q=` server filter when history outgrows what the phone holds.
- **Swipe-to-reply** — quoted replies threading onto a specific agent message
  (the v0.2 "quoted/threaded replies" item, now with its gesture).
- **Pinned conversations** — pin the agent doing today's critical work to the
  top of the list; sort pinned-first, then recency (client-side preference).

## Cross-crew connector (still separate, prerequisite for native Magnus↔bridge)

For a Magnus agent to be a bridge *contact* (addressable via `bridge send --to`)
without leaving Emacs: a **register-only mode** whose delivery path is a Magnus
nudge (emacsclient) and whose replies tail that agent's session JSONL. Until
then, cross-crew messages ride Magnus's own wire (as quick-wolf ↔ Vint did).
