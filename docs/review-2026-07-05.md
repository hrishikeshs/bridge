# bridge — security & correctness review (2026-07-05)

Five independent adversarial reviewers (opus, read-only) swept the daemon and PWA:
trust boundary ×2, command-execution path, daemon concurrency, PWA client. This is
the deduplicated, severity-ranked, **triaged** result. Each item has a stable ID,
the owning file(s), the concrete failure, the fix, and a verdict:

- **FIX** — confirmed real, hand to an implementer.
- **DECISION** — real, but the resolution is Hrishi's call (not an auto-fix).
- **DEFER** — real, but only bites outside Hrishi's current setup / low value now.
- **REJECT** — false positive, dismissed with reasoning.

The threat model that weights everything: a phone message becomes keystrokes in a
live Claude Code agent, so anything that (a) grants auth, (b) misroutes input, or
(c) silently drops a command, is severe.

## CRITICAL

### C1 — Pairing code is brute-forceable  · FIX · `trust.go`, `serve.go`
Corroborated by **3 independent reviewers**. 6-digit code, **no rate limit, no
attempt cap, a wrong guess is free**, code stays valid the full 10-min window and
is a static target the whole time (`tryPair` trust.go:170-188 only clears on
success). `/api/pair` is reachable with just the identity gate, no token
(serve.go:175). ~1,700 guesses/sec covers the whole 10⁶ space inside one pairing
window; on loopback/LAN that's trivial. Win → long-lived device token → type
commands into any agent = RCE as the user. Compounding: `issuePairingCode`
(trust.go:155-166) has modulo bias and **ignores the `rand.Read` error** (yields
`000000` on RNG failure); comparison is non-constant-time.
**Fix:** invalidate the code after N≈5 failed attempts (force re-issue); widen it
(≥8 digits or a word passphrase) and/or add a global token-bucket on `/api/pair`;
`crypto/subtle.ConstantTimeCompare`; reject-sample the modulo bias and panic on
`rand.Read` error (match `randomHex`); reconcile TTL to the shorter documented
value.

### C2 — Unvalidated name → tmux target-grammar misroutes messages & approvals to the WRONG live agent  · FIX · `session.go` (+ `registry.go`)
**Only the command-execution reviewer caught this — reproduced against real tmux.**
`name` becomes the routing key `bridge:<name>` in every `send-keys`/`capture-pane`
(session.go:330, 81, 109), but tmux resolves the window part of a target by its
*grammar*, which disagrees with `tmuxAlive`'s exact-string match:
- `--name 1` (numeric) → `send-keys -t bridge:1` resolves `1` as **window index 1**
  → every message and approval for agent `1` lands in a *different* agent's pane
  (reproduced: "misrouted to BOB"). Permanent.
- `--name +` / `-` (relative) → resolves relative to the active window.
- Because `/api/approve` uses the identical target, a whitelisted approval
  keystroke goes to the wrong agent — approving a prompt the human never saw.
- NOT argv flag-injection (blocked: name follows `-n`) and NOT shell (no shell).
  It's tmux *target-string* interpretation. `=`-prefixing does **not** fix numeric
  names.
**Fix:** (1) validate name at connect (`runConnect` + `handleLocalConnect`):
`^[A-Za-z][A-Za-z0-9_-]{0,30}$`, reject all-digit / relative / special / whitespace.
(2) Route by **window id**: capture `@N` at creation (`new-window -P -F
'#{window_id}'`) and store *that* as `TmuxTarget` — immune to the grammar.
**This is the same code path as the names feature (auto-generation + uniqueness),
so the fix and the feature land together.**

## HIGH

### H1 — Failed send + retry silently swallows the message  · FIX · `serve.go`
`handleSend` records `client_id` as seen **before** attempting delivery
(serve.go:440). If `deliverToSession` fails (tmux busy/target hiccup) → 500 → the
phone retries the same `client_id` → now "duplicate" → `200 {ok:true}`, **delivery
never attempted.** Message gone, phone shows it sent. `handleUpload` is worse — the
attachment file is already on disk, so the retry drops the message *and* orphans
the file. The dedup invariant is backwards: a repeated id is only safe to ack if
the *first* attempt succeeded.
**Fix:** record the `client_id` only *after* durable success (delivered to session
or queued to mailbox). Split into check-up-front / commit-on-success.

### H2 — Two contacts in the same directory share one tailed session file → reply cross-talk  · FIX · `session.go`
`pollReplies` resolves the session file by **directory** (`currentSessionFile(c.Directory)`,
session.go:159), ignoring the contact's own `SessionID`. Two agents rehomed from
the same repo dir (legal — identity is name+directory) both tail the newest
`.jsonl`: one agent's replies appear under *both* contacts' threads, the other's
are never tailed. Both also `SetSession` to the same id, so `BySession` routes
permission-attention cards to the wrong thread.
**Fix:** select the session file by the contact's own `SessionID`
(`projectDir(dir)/<SessionID>.jsonl`), fall back to newest only when empty; guard
`SetSession` against adopting an id already held by another live contact.

### H3 — A compromised agent can self-approve its own permission prompt  · DECISION · `protocol.md`
Runs as the same UNIX user as the daemon, so it can `cat ~/.bridge/tokens.json`,
POST `/api/approve` with the real device token, and type `1` into its own pending
`Bash(rm -rf …)` dialog — no human. `protocol.md` §7.6 promises "the bridge extends
the human loop, never removes it." **That promise is stronger than a same-uid
architecture can keep.** Largely *inherent* — the file-perm boundary (0600) stops
other users, not the very process it's meant to contain. Not cleanly fixable
without running agents as a separate uid or an out-of-band approve confirmation.
**Decision for Hrishi** (his honesty-over-theater value applies directly):
(a) reword the doc to state the real boundary [recommended, ships now], (b) also
add a real mitigation (out-of-band approve / separate uid — bigger), or (c) leave
the wording.

## LOW  (mostly DEFER; cheap safe ones marked FIX)

- **L1 — Default config accepts *any* tailnet login + loopback identity header is forgeable by other local users.** · DEFER (for Hrishi's setup) · `trust.go`
  Real, but only bites on a **shared tailnet or multi-user host**. Hrishi's Mac is
  solo/single-user → contained today. **Do NOT auto-flip to fail-closed** — that
  would stop the live daemon from serving until `allowed_logins` is set. Fix before
  a stranger runs bridge: default-deny + pin the owner's login at first `expose`.
- **L4 — Control-byte passthrough in `send-keys`** · FIX · `serve.go` — `formatInbound` strips only `\n\r`; ESC/arrows pass literally (esp. agent→agent). Sanitize to printable + space.

- **L6 — SW `push` handler throws on `null` JSON** · FIX · `pwa/sw.js` — guard `d`.
- **L2 — Pairing TTL doc drift** (code 10 min, docs/PWA say 2) · FIX (folded into C1).
- **L3 — Non-constant-time compares** (pairing code, local token) · FIX (folded into C1).
- **L7** attachments never pruned · DEFER. **L8** SSE broadcast reorder (self-heals) · DEFER. **L9** `loadPushSubs` unlocked (latent) · DEFER. **L10** `looksLikePrompt` false pos/neg (external UI text) · DEFER. **L11** audit log floodable/no rotation · DEFER. **L12** `ensureBridgeDir` doesn't chmod an existing dir · FIX (one line). **L13** device tokens unbound/non-expiring · DEFER.

## Verified clean (reassuring — corroborated across reviewers)

- **No command injection** — every subprocess is `exec.Command` argv-form, no shell;
  a crafted `--name` is inert in tmux *args* (the risk is C2's target grammar, not args).
- **No send-keys keystroke injection** — messages sent with `-l --` (literal) + a
  separate `Enter`; `formatInbound` pre-flattens newlines. Can't inject a second line.
- **Path traversal closed** (`fs.ValidPath` + `embed.FS`); **approve keys** a closed
  whitelist gated on hook-attested `PromptOpen`; **ReDoS** N/A (RE2).
- **PWA is XSS-safe today** — all render via `textContent`/`createTextNode`; linkifier
  enforces http(s) + `rel="noopener noreferrer"`. Device token not JS-readable.
- **No registry races** — `Contact.copy()` fully detaches (scalar fields); no lock
  reentry/deadlock; SSE lifecycle leak-free; `sendPush` status classification correct
  (a failed send is never counted delivered); `newID` is a proper crypto/rand UUID.
