# bridge — review of PR #1 (`review-fixes` → `master`, 2026-07-05)

*Second-order review: this reviews the PR that landed the fixes from
`docs/review-2026-07-05.md`. Eight adversarial finder angles were run over the
full diff (line-by-line, removed-behavior, cross-file tracer, reuse,
simplification, efficiency, altitude, conventions); every finding below was then
verified against the actual source — and one against the live filesystem — so
these are confirmed, not speculative.*

**PR under review:** *Harden daemon + PWA from 5-way security review; add names
feature.* 868 +/100 −, 11 files. `go build` + `go vet` + `gofmt` clean; smoke
suite **29/29** on this head.

---

## Verdict

The security posture genuinely improved and the PR is safe to merge **as the
direction**. But several of the *new* hardening paths have gaps, and two items
are regressions only session-context catches. Recommend fixing the TTL revert
and findings #1–#3 before this becomes the canonical `master`, since #1 and #2
leave the two **headline** fixes (C2 routing, the names feature) half-open.

---

## The most important catch — a silent regression

### R0 — Pairing TTL was reverted 10 min → 2 min · `trust.go:19`
The PR frames this as "reconcile the doc drift / shrink the brute-force window."
But the TTL was **deliberately moved to 10 minutes earlier the same day** because
2 minutes forced ~8 code re-entries during real phone setup (recorded in the
keystone session notes). The security rationale doesn't hold: the source review
doc itself says *"this attempt-cap — not the digit count — is what defeats the
brute-force."* With the new 5-attempt cap, a 10-minute window is exactly as safe
as a 2-minute one (5 guesses out of 10⁶ either way) — and 2 min is now *more*
punishing, because fat-fingering the code 5× also locks it.

**Fix:** TTL back to 10 min; update `docs/protocol.md:125` and
`pwa/index.html:25` *up* to 10 (they still say 2 — that was the original drift,
never the intended value).

## Confirmed correctness bugs (most severe first)

### 1 — The marquee C2 fix is CLI-only; the daemon boundary is still porous · `serve.go:675` (`handleLocalConnect`)
Name validation (`nameConnectRe`) and `@N` routing live only in `runConnect`.
The shared daemon entry point passes `req.Name` and `req.TmuxTarget` verbatim
into `registry.Connect`. A numeric name `1` + legacy target `bridge:1` sails
straight through — and the **rollback binary `~/bin/bridge.bak`** and
`test/smoke.sh` both speak exactly this protocol. The source review doc
explicitly prescribed validating in *both* places; only one got it.
**Fix:** move the regex + target canonicalization into `handleLocalConnect` (or
`registry.Connect` itself) so no producer can bypass it.

### 2 — The names feature hijacks a live agent's window on collision · `session.go` (`ensureWindow`)
`tmuxWindowID(name)` returns *any* existing same-named window, including one
owned by a different live agent → no new `claude` is spawned, the daemon
suffixes the newcomer to `name-2` but stores the **hijacked `@N`**, and the CLI
then renames the *original's* window. Every message/approval to `name-2` is typed
into the original agent's pane — a C2-style misroute *through the fixed code
path*. The collision case the uniqueness suffix exists for is the one it breaks.

### 3 — H1's dedup split reintroduces a TOCTOU double-send · `serve.go:483`
`seenClientID` (read) → `deliverToSession` → `commitClientID` (write) is no
longer atomic. The PWA aborts at a 10s timeout and retries; if the original
handler is still in `deliverToSession` (slow tmux), both requests pass
`seenClientID` and both `send-keys` — a command or a `yes`/`1` approval typed
**twice** into the agent. The old `duplicateClientID` was atomic and precluded
exactly this.
**Fix:** one helper that marks in-flight under `seenMu`, runs delivery, commits
on success / releases on failure.

### 4 — `skipOversizedLine` defeats the memory cap it exists to enforce · `session.go:379`
It uses `bufio.ReadBytes('\n')`, buffering the *entire* oversized line — which
by definition already exceeded the 8 MB `maxJSONLLine` cap — just to measure its
length, and re-buffers every 2s while the line is unterminated. A
multi-hundred-MB tool-result line spikes RSS by its full size each poll →
possible OOM.
**Fix:** read fixed 64 KB chunks, counting bytes to the next `\n`, retaining
nothing.

### 5 — `FlushMailbox` can drop an undelivered message at the cap · `registry.go:347`
It delivers `q[0]`, releases the lock, re-locks and slices `q[1:]` — but the new
`Queue` drop-oldest (`maxMailbox=100`) can shift the head in between, so `q[1:]`
discards a message that was *never delivered*. The inline `// Queue only appends`
comment is false precisely at the cap.
**Fix:** re-find the delivered message by identity before removing it, or
snapshot the head under one lock.

### 6 — A suffixed agent posts to the wrong thread · `session.go:500`/`553`
`BRIDGE_CONTACT` is baked into the window env with the *pre-suffix* name;
`runSend` reads it as its own identity. If the daemon suffixed the agent to
`name-2`, its `bridge send` still self-identifies as `name`, and `Resolve(name)`
returns the *other* live agent — outbound messages and pushes attributed to the
wrong contact. Same root as #2: identity is mutated after the env/window are
fixed.
**Deeper fix:** route by immutable contact **id** (`BRIDGE_CONTACT=<id>`),
reconcile the display name daemon-side.

### 7 — DNS rebinding defeats the M13 SSRF guard · `push.go:100`
`validPushEndpoint` resolves the host once at subscribe time (and fails *open* on
lookup error, accepting NXDOMAIN); `sendPush` later dials with the default client
and no guard. Subscribe with a public IP, then rebind to `127.0.0.1` → every push
POSTs to the internal address and `audit()` logs the response body.
**Fix:** enforce `isInternalIP` at dial time via a custom `Transport.DialContext`
passed through `webpush.Options.HTTPClient` — where the request is actually made.

### 8 — Compaction can double-write an event → duplicate bubble after restart · `events.go:131`
`appendHistory` now runs outside `eventsMu`; a compaction rewrite (from a ring
already holding event N) can interleave with N's own deferred append, so N lands
on disk twice, and `loadHistory` sorts by id but never dedups. Low severity —
rare interleave, cosmetic, post-restart only.
**Fix:** a one-line id-dedup in `loadHistory`.

### Design tension — the C1 attempt-cap adds a pairing lockout-DoS · `trust.go`
Any identity-gated-but-unpaired tailnet peer can fire 5 wrong guesses to
invalidate every code you issue. Contained on a solo tailnet today; if bridge
ever goes multi-user, cap per-source rather than globally (or lock only after N
*distinct* device ids).

## The scary one that isn't real — investigated and refuted

**Four separate finders claimed the reply loop is "permanently broken"** because
`sessionFileFor` (`session.go:359`) pins tailing to the pre-rehome `SessionID`,
reasoning from the `registry.go:13` comment *"churns on resume."*

**Refuted empirically.** This exact session was rehomed via
`claude --resume 988f4912`, and `988f4912.jsonl` is on disk at 21.5 MB with a
mtime of minutes ago — still the live, growing file. `--resume` **preserves** the
id on this Claude Code version, so pinning by `SessionID` is correct and the
reply loop works (as proven live).

**But there's a legitimate fragility:** the new code dropped the old code's
self-heal (fall back to the newest `.jsonl`), so if a future CC version *does*
fork ids on resume, replies would break with no recovery — and the misleading
`registry.go:13` comment is what sent four reviewers down this path.
**Cheap insurance:** keep the newest-file fallback when the pinned file is
missing or stale, and fix that comment.

## Cleanup (lower priority, real)

- **Dead code:** `TakeMailbox` + `Requeue` (`registry.go:358-378`) have zero
  callers now — and they're the unsafe delete-then-lose API that M6 was written
  to kill. Delete them before someone reaches for the obvious-sounding name and
  reintroduces the batch-loss bug.
- **`randIndex`'s modulo rides a comment-invariant** (256 % 16 == 0,
  `registry.go`). Pad a wordlist to 17 and you silently reintroduce the exact
  modulo bias C1 just fixed. A shared reject-sampling `randInt` next to
  `randomHex` makes that impossible — and folds in the near-duplicate rand blocks
  (`pairingDigits`).
- **Minor duplication** (all optional; each shrinks the "fix it in one place"
  surface):
  - `seenClientID`/`commitClientID` duplicate one ring-scan.
  - `tmuxAlive`'s `@`-branch and `tmuxWindowID` are two tmux parsers to keep in
    sync (and they already diverge on scope: `-a` vs `-t bridge`).
  - the `.corrupt` rename is copy-pasted between `loadRegistry` and `loadHistory`.
  - `Emit`'s variadic `clientID` is a faux-optional param (2 of 12 call sites use
    it); an explicit param or an `EmitSent` wrapper is clearer.

## Suggested handoff

Three disjoint workstreams, same partition style as the source review:

- **A — Trust & push** · `trust.go`, `push.go` · R0 (TTL + docs), #7 (dial-time
  SSRF), lockout-DoS note, `randInt` consolidation.
- **B — Session & registry** · `session.go`, `registry.go` · #1 (daemon-side
  validation), #2 (ensureWindow collision), #4 (chunked skip), #5 (FlushMailbox
  cap race), #6 (id-based identity), reply-loop fallback, dead-code deletion.
- **C — Serve & events** · `serve.go`, `events.go` · #3 (atomic client_id
  lifecycle), #8 (loadHistory dedup).

`#1` and `#2` leave the two headline fixes (C2 routing, the names feature)
half-open, so they + R0 are the merge-blockers; the rest can follow on the
branch.
