# Bridge cross-cutting review — 2026-07-06

Six-lens parallel review (message-loss, concurrency, state/liveness, security,
PWA-client, resilience), each finding adversarially verified by an independent
refuter reading the code as written. 45 agents, ~1.8M tokens.

**46 findings: 4 critical, 14 high, 13 medium, 6 low, 9 design notes.**
38 CONFIRMED, 1 REFUTED, 7 design recommendations.

The headline: **all four criticals are one root cause.** The delivery pipeline
(`flushMailbox` → `tmuxDeliver` → `send-keys` + Enter) types into a pane
without first verifying the pane is (a) running Claude Code and (b) not showing
a permission dialog. Every critical is a variation on typing into the wrong
thing. One guard closes all four.

---

## CRITICAL — the unguarded-pane family (one root cause, four faces)

The coalescer's `PromptOpen` check (coalesce.go:83) was the *only* guard against
typing into an open dialog, and it is (a) bypassed by every non-coalesce
delivery path and (b) an attestation flag that lags or is wiped, not pane truth.

- **C1 — flush into the pre-Claude bare shell** (serve.go:778, msg-loss + concurrency).
  `runConnect` creates a shell-only tmux window, POSTs `/local/connect` (which
  `flushMailbox`es synchronously), *then* launches Claude. So queued mail is
  send-keys'd into a bare `zsh` — which the shell tries to execute — and then
  `respawn-pane -k` kills the shell, erasing it. Mail is dropped as "delivered."
  (Shell-execution is mostly defanged by the `[From X (phone)]:` prefix being a
  zsh parse error, but the silent message loss is unconditional.)

- **C2 — revive flush into an open dialog** (session.go:192, msg-loss/state/resilience).
  On daemon restart, `loadRegistry` and `Connect` both reset `PromptOpen=false`;
  the reconcile revive branch calls `flushMailbox` directly. A window that
  outlived the restart with a permission dialog open gets the held mail typed in
  — **Enter selects the highlighted option**, silently approving a command the
  user never saw, and the message text is swallowed by the modal.

- **C3/C4 — the same, via the restart and reconnect paths** (serve.go:789).
  `flushMailbox` has no prompt check anywhere; both revive callers bypass the
  one in `coalesceDeliver`. This is *exactly* the 3-hour-sleep field failure,
  now with unauthorized approval on top.

**The fix (one guard, shipped this round):** make `flushMailbox` itself capture
the pane and refuse to deliver unless the pane is Claude (`paneSessionID != ""`)
and shows no dialog (`!looksLikePrompt(capturePrompt(c))`), re-arming instead of
dropping. Drive retries from the reconcile loop so deferred mail is never
stranded. On revive, re-detect an open dialog and re-raise the attention card
instead of clearing it. → closes C1–C4 + H1, H2, H5 + L1, L2 below.

## MEDIUM (13) — themes

- **Prompt-state lag** (serve.go:597, :623; session.go:317): `PromptOpen` stays
  true seconds after approve/interrupt — a second tap types `1`+Enter as a fake
  user message; health flaps `prompt`→`working`. → optimistic `SetPrompt(false)`
  after a daemon-issued keystroke.
- **Switchboard offline = 409** (serve.go:913): durably queued but reported as
  failure → sender retries → dupes. → 200 `{queued:true}`.
- **maxMailbox silently drops acked messages** (registry.go:346) held live for
  coalescing. → audit the drop; sender-aware policy.
- **mtime tie-break survives in one more spot** (session.go:428) — *already
  fixed today in `58b41aa`* but flagged independently; verify no other caller.
- **Push races** (push.go:311): a stale "✓ handled" can replace a newer "needs
  you"; the 3s debounce can swallow a real prompt. → per-contact push sequencing.
- **Chain scan blocks the reconcile goroutine** (session.go:434): a
  hundred-MB directory scan stalls prompts/revives/relay for *everyone*. → bound
  the scan / run per-contact with a deadline.
- **PWA notification hygiene** (app.js:428, :1610, :1515):
  `clearDeliveredNotifications` closes *other* agents' still-live prompts;
  page-scoped `maybeNotify` escapes all supersede logic + double-notifies on
  desktop; `dropPendingEcho` text-fallback can splice the wrong queued message.
- **Stale-lockfile no self-heal** (session.go:939); **dead SSE after wake**
  (app.js:270).

## DESIGN NOTES — the resilience slate (answering today's 2-hour blackout)

Ranked by impact-per-effort. Two marked **ship this week**.

1. **Wall-clock gap watchdog** ⭐ *(ship this week)* — reconcile loop tracks
   `last := now`; a gap >90s means the Mac slept. Emit a "back online, asleep
   HH:MM–HH:MM" system event + push, and re-audit every session file (mtimes are
   3h stale). Turns a silent blackout into a notification. (session.go:168)
2. **launchd supervision** ⭐ *(ship this week)* — `bridge install-daemon` writes
   a `KeepAlive` LaunchAgent: survives reboot, restarts on crash, rotates logs.
   Retires the `nohup` deploy dance that *caused* today's outage (a deploy froze
   on its own permission prompt). (serve.go:107)
3. **Phone-side truth** — `/api/status` returns server clock + daemon start +
   last-gap; the PWA shows "Mac asleep since 10:02" instead of a generic
   "unreachable." (app.js:359)
4. **Frozen-agent escalation** — a prompt open >5/15/30 min re-rings the phone
   (pushes ride APNs, they work off-tailnet). Auto-answer is explicitly a trap —
   don't. (serve.go:829)
5. **Daemon self-caffeinate** — `caffeinate -w <pid>` child, honest README that
   *only* `pmset disablesleep` survives a closed lid. (serve.go:107)
6. **Persist tail offsets + attnPushed** — a restart neither replays nor
   swallows agent output, and stale lock-screen demands resolve. (session.go:163)
